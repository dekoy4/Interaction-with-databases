## Часть №2

### Ответы на вопросы:

#### Почему нет JOIN-ов на лету

Из-за колоночного способа хранения данных в Сlickhouse операции склеивания таблиц через JOIN в рамках запроса крайне неэффективны. Индексирование происходит не построчно, а по блокам, содержащим набор строк, и поиск определенной строки для склеивания происходит путем полного анализа блока с перебором всех строк. Таким образом Увеличивается время поиска и затраты по памяти (из-за загрузки больших блоков в память).  

#### Почему избыточность данных компенсируется сжатием

Потому что при колоночном хранении алгоритмы сжатия оптимизируют хранение повторяющихся значений в столбцах одним из алгоритмов (через словарное кодирование или алгоритмы оптимизации числовых последовательностей). Таким образом вся избыточность нивелируется сжатием повторяющихся значений.

#### Почему LowCardinality заменяет справочные таблицы

Справочные таблицы хранят связку индекса и значения. LowCardinality представляет собой словарь с ключом - индексом и значением из столбца, находясь при этом локальном в том же файле, где и сам столбец. Это позволяет производить операцию чтения и подстановки локально, без необходимости обращаться в другую (справочную таблицу).

## Часть №6

Выполнение команды:

```sql
SELECT
    column,
    formatReadableSize(sum(data_compressed_bytes)) AS compressed,
    formatReadableSize(sum(data_uncompressed_bytes)) AS uncompressed,
    round(sum(data_uncompressed_bytes) / sum(data_compressed_bytes), 2) AS ratio
FROM system.parts_columns
WHERE database = 'test_db' AND table = 'orders_flat' AND active
GROUP BY column
ORDER BY sum(data_uncompressed_bytes) DESC;
```

1. В данном случае сжатие одинаково для всех столбцов независимо от типа данных (несмотря на разные размеры сжатых и несжатых данных для каждой колонки в каждой партиции). Наибольшее сжатие должны иметь столбцы order_status, region и category (LowCardinality) за счет хранения только индексов словаря (значения словаря - искомые строки) и частых повторений строк.
2. LowCardinality меняет формат хранения на словарное кодирование, где данные разбиваются на слова и заменяются на индексы в словаре. Благодаря хранению повторяющихся индексов словаря вместо исходных данных значительно повышается степень сжатия.
3. ORDER BY влияет на физический порядок расположения блоков данных на диске, в то время как алгоритмы сжатия данных имеют ограниченный размер скользящего окна. Таким образом в скользящее окно алгоритма попадет больше одинаковых данных, подходящих под параметры сжатия и оптимизации, при близком расположении блоков данных с одинаковыми данными или имеющие повторяющийся паттерн.

## Часть №7 

| Характеристика | OLTP (PostgreSQL) | OLAP (ClickHouse) |
|----------------|-------------------|-------------------|
| **Модель хранения** | Строковая | Колоночная |
| **Типичный запрос** | `INSERT`, `UPDATE`, `DELETE`, `SELECT` по ключу | Агрегатные запросы: `SUM`, `GROUP BY`, `COUNT` |
| **Нормализация** | Высокая (3НФ) | Плоские повторяющиеся таблицы, повторяющиеся данные |
| **Транзакции** | Полная поддержка ACID, `BEGIN`/`COMMIT`/`ROLLBACK` | Нет транзакций (возможно ограниченная поддержка) |
| **Вставка** | Построчная, поддерживаются `BEGIN`/`COMMIT` | Пакетная, лучше вставлять пачками (гранулами) |
| **Обновление/удаление** | Полная быстрая поддержка `UPDATE`/`DELETE` | Ограниченая, дорогая операция |
| **Масштабирование** | Вертикальное, горизонтальное через шардинг | Горизонтальное нативное, шардинг из коробки |
| **Типичный use case** | Интернет-магазины, платёжные системы, обработка заказов | Аналитика, логи, метрики |
